Process-to-Process Delivery — Overview

What is the Transport Layer? ما هي طبقة النقل؟

The transport layer is responsible for process-to-process delivery — delivering a packet (or part of a message) from one process on one host to another process on another host. This is different from network-layer (host-to-host) delivery.

Application A
Process on Host 1
→
Transport Layer
Process-to-Process
→
Application B
Process on Host 2
Connectionless (UDP)
Each packet treated independently. No setup. Fast but unreliable.
Connection-Oriented (TCP)
Virtual connection established first. Reliable, ordered, error-controlled.
Arabic Explanation
طبقة النقل مسؤولة عن التوصيل من عملية إلى عملية (Process-to-Process) — وليس فقط من جهاز لجهاز.
الفرق الجوهري: طبقة الشبكة (Network Layer) توصل بين الأجهزة (Host-to-Host). طبقة النقل توصل بين البرامج/العمليات داخل تلك الأجهزة.
مثال: جهازك يعمل Chrome (منفذ 80) وZoom (منفذ 3478) في نفس الوقت — طبقة النقل تُميّز بين البيانات الخاصة بكل برنامج.
Multiplexing & Demultiplexing الإرسال والاستقبال المتعدد
Multiplexing (at sender):

Multiple processes send data through one transport layer. The transport layer combines them and sends over the network.

Demultiplexing (at receiver):

The transport layer receives data from the network and directs each packet to the correct process based on the port number.

Arabic
Multiplexing (جمع/دمج): في جهاز المرسل، عمليات متعددة ترسل بياناتها عبر طبقة نقل واحدة — يتم دمجها معاً للإرسال عبر الشبكة.
Demultiplexing (توزيع): في جهاز المستقبل، طبقة النقل تستقبل الحزم وتوزّعها للعملية الصحيحة بناءً على رقم الـ Port.
تخيّلها: Mux = بريد يُجمع من عدة أشخاص. Demux = ساعي البريد يوزّع على الأشخاص الصحيحين.
Client / Server Paradigm نموذج العميل/الخادم

Processes communicate in a client/server relationship:

Server
Runs continuously. Waits for requests on a well-known port (0–1023). Example: Web server on port 80.
Client
Initiates request. Assigned a temporary (ephemeral) port by the OS (49152–65535). Example: Your browser.
Arabic
السيرفر يعمل باستمرار وينتظر على منفذ معروف (well-known port). العميل يبدأ الاتصال ويستخدم منفذاً مؤقتاً تُعطيه إياه نظام التشغيل.
مثال: سيرفر HTTP يعمل على المنفذ 80 دائماً. متصفحك يختار منفذاً عشوائياً (مثلاً 51234) لكل اتصال.

Socket Address عنوان الـ Socket

A socket address uniquely identifies an endpoint of communication by combining the IP address and port number:

Socket Address Format
Socket Address = IP Address : Port Number Examples: 192.168.1.5 : 80 → HTTP server on this host 10.0.0.1 : 52100 → Client process (ephemeral port) To uniquely identify a connection (socket pair): (Source IP : Source Port) ↔ (Destination IP : Destination Port)
① Key Rule
IP address alone identifies the host. Port number alone identifies the process. Together (socket address) they uniquely identify a communication endpoint.

UDP — User Datagram Protocol

بروتوكول UDP — بروتوكول مخطط البيانات
UDP Overview & Characteristics نظرة عامة على UDP وخصائصه

UDP is a connectionless, unreliable transport protocol. It adds process-to-process communication on top of IP, but does not add reliability, flow control, or error correction.

Connectionless
No handshaking before sending. Each datagram is independent.

Unreliable
No delivery guarantee. No order guarantee. No duplicate detection.

Fast & Lightweight
Small 8-byte fixed header. Minimal overhead. Low latency.

Best uses for UDP:

  • DNS — short queries, speed matters
  • VoIP / Video calls — real-time, tolerate loss
  • Online gaming — latency critical
  • Streaming (live) — old frames useless
  • DHCP, TFTP, SNMP
  • Broadcast / Multicast — TCP can't do these
Arabic Explanation
UDP = بروتوكول بسيط وسريع بدون ضمانات.
Connectionless: لا يوجد تأسيس اتصال مسبق — يرسل مباشرة بدون مصافحة.
Unreliable: لا يضمن الوصول، لا يضمن الترتيب، لا يكشف التكرار.
متى نستخدم UDP؟ عندما تكون السرعة أهم من الدقة — الصوت والفيديو المباشر، الألعاب الإلكترونية، DNS. خسارة حزمة واحدة في مكالمة صوتية أفضل من التأخير الذي يسببه TCP.

Solved Example — Reading a UDP Header in Hex مثال محلول — قراءة UDP Header
Problem
The following is a dump of a UDP header in hexadecimal: CB84 000D 001C XXXX
Find: (a) Source port (b) Destination port (c) Total length (d) Data length (e) Client→Server or Server→Client? (f) What client process?
Solution
Reading hex, each group of 4 hex = 16 bits = one field:
Hex Breakdown
CB84 = 1100 1011 1000 0100 = 52100 decimal → Source Port 000D = 0000 0000 0000 1101 = 13 decimal → Destination Port 001C = 0000 0000 0001 1100 = 28 decimal → Total Length (bytes) XXXX = Checksum (given or computed)
  1. Source Port = (CB84)₁₆ = 52,100
  2. Destination Port = (000D)₁₆ = 13
  3. Total Length = (001C)₁₆ = 28 bytes
  4. Data Length = 28 − 8 = 20 bytes
  5. Destination port = 13 (well-known) → packet is from Client → Server
  6. Port 13 = Daytime Protocol
Arabic — طريقة الحل
كيف تقرأ UDP Header من Hex:
— الـ 4 أرقام الأولى = Source Port. الـ 4 الثانية = Destination Port. الـ 4 الثالثة = Total Length. الـ 4 الرابعة = Checksum.
— حوّل كل مجموعة من Hex إلى Decimal.
— Data Length = Total Length − 8 (حجم الـ Header دائماً 8 بايت).
— إذا الـ Destination Port من 0-1023 (well-known) → الحزمة من Client لـ Server.
— إذا الـ Source Port من 0-1023 → الحزمة من Server للـ Client.
سؤال شائع في الامتحانات: تحويل Hex لـ Decimal وتحديد اتجاه الحزمة ونوع الخدمة.
UDP Checksum Calculation حساب Checksum في UDP

The UDP checksum covers a pseudo-header + UDP header + data. The pseudo-header is NOT transmitted — it's used only for checksum calculation.

Pseudo-Header Contents (used for checksum only)
| Source IP Address (32 bits) | | Destination IP Address (32 bits) | | Zeros (8) | Protocol=17 (8 bits) | UDP Length (16 bits) | | — UDP Header — | — UDP Data — | Steps: 1. Divide the above into 16-bit words 2. Add all words together (binary addition with wraparound carry) 3. Take 1's complement of the sum = Checksum At receiver: add all words including checksum → result should be all 1s (FFFF)
Arabic
الـ Checksum يغطي: Pseudo-Header (عناوين IP + بروتوكول + الطول) + UDP Header + البيانات.
الـ Pseudo-Header لا يُرسل — يُستخدم فقط في حساب الـ Checksum للتحقق من أن الحزمة وصلت للجهاز الصحيح.
طريقة الحساب: تقسيم كل شيء إلى كلمات 16 بت، جمعها، ثم أخذ مكمل الواحد (One's Complement).
في المستقبل: إذا جمع كل شيء بما فيه الـ Checksum وكانت النتيجة FFFF → لا يوجد خطأ.
UDP Queues & Operation قوائم الانتظار وآلية عمل UDP
At the Client:

When a process starts, it requests a port from the OS. Some implementations create both incoming and outgoing queues. If the incoming queue overflows — packets are silently dropped (no notification).

At the Server:

The server requests incoming and outgoing queues using its well-known port on startup. These queues remain open as long as the server runs. Multiple clients share the same server queue.

Queue Overflow: UDP has no flow control. If the receiving queue is full when a new datagram arrives, the datagram is silently discarded — no error message is sent to the sender.
Arabic
UDP يعمل بقوائم انتظار (Queues) لكل عملية. في جهة العميل: يطلب منفذاً من نظام التشغيل ويعمل قائمة للإرسال والاستقبال.
في جهة السيرفر: قائمة مفتوحة طالما السيرفر يعمل — عملاء متعددون يتشاركون نفس القائمة.
الخطر: إذا امتلأت القائمة → الحزمة تُحذف بصمت تام بدون إشعار للمُرسل. لا يوجد Flow Control في UDP.

TCP — Basics & Core Concepts

TCP — المفاهيم الأساسية
TCP Overview نظرة عامة على TCP

TCP (Transmission Control Protocol) is a connection-oriented, reliable transport protocol. It creates a virtual connection between two TCPs before sending any data.

Connection-Oriented
Must establish a virtual connection (3-way handshake) before data transfer.
Reliable
Guaranteed delivery, no loss, no duplication, no corruption.
Ordered
Data arrives in the same order it was sent — TCP reassembles if needed.

TCP provides:

  • Stream delivery — data seen as continuous byte stream (not messages/packets)
  • Byte numbering — every byte is assigned a number
  • Sequence numbers — track position in the byte stream
  • Acknowledgment — receiver confirms received bytes
  • Flow control — receiver controls sender's rate (sliding window)
  • Error control — detect and retransmit lost/corrupted segments
  • Congestion control — avoid overloading the network
Arabic Explanation
TCP بروتوكول موثوق ومتصل. يُنشئ اتصالاً افتراضياً قبل إرسال أي بيانات.
Stream Delivery: TCP يرى البيانات كتدفق مستمر من البايتات — مثل نهر مائي مستمر. لا يوجد حدود للرسائل في الـ TCP Layer.
الميزات الرئيسية: ترقيم البايتات، أرقام التسلسل، التأكيدات، التحكم في التدفق، اكتشاف الأخطاء وإعادة الإرسال، التحكم في الازدحام.
متى نستخدم TCP؟ عندما يكون الموثوقية مطلوباً — نقل الملفات (FTP)، صفحات الويب (HTTP)، البريد الإلكتروني (SMTP).
Byte Numbering & Sequence Numbers ترقيم البايتات وأرقام التسلسل
Byte Numbering: TCP numbers every byte of data. The starting number is randomly generated (ISN — Initial Sequence Number) for security reasons.
① Sequence Number Rule
The value in the sequence number field of a segment defines the number of the first data byte contained in that segment.
② Acknowledgment Number Rule
The value in the acknowledgment field defines the number of the next byte the receiver expects to receive. The acknowledgment is cumulative — it confirms all bytes up to that point.
Example — Sequence & ACK Numbers
Suppose host A starts with ISN = 1000 and sends 100 bytes:
→ Segment 1: seq=1000, data bytes 1000–1099
→ Segment 2: seq=1100, data bytes 1100–1199
Host B receives Segment 1 → sends ACK: ack=1100 (next byte expected)
Host B receives Segment 2 → sends ACK: ack=1200
Arabic
ترقيم البايتات: TCP يرقّم كل بايت. الرقم الأولي (ISN) عشوائي للأمان.
Sequence Number: يمثل رقم أول بايت في الـ Segment الحالي.
Acknowledgment Number: يمثل رقم البايت التالي الذي يتوقع المستقبل استلامه — وهو تراكمي (Cumulative) أي يؤكد استلام كل البايتات حتى هذا الرقم.
مثال: المرسل عنده ISN=1000 وأرسل 100 بايت → Seq=1000. المستقبل يرد بـ ACK=1100 (يتوقع البايت التالي رقم 1100).

Three-Way Handshake Summary
Step 1 (Client → Server): SYN, seq=x [SYN consumes seq x] Step 2 (Server → Client): SYN+ACK, seq=y, ack=x+1 [SYN+ACK consumes seq y] Step 3 (Client → Server): ACK, ack=y+1 [ACK with no data: no seq consumed] After Step 3: Connection is ESTABLISHED. Data transfer can begin.
Critical Rules — Exam Favorite
  • A SYN segment CANNOT carry data, but it DOES consume one sequence number.
  • A SYN + ACK segment CANNOT carry data, but it DOES consume one sequence number.
  • An ACK segment carrying NO data DOES NOT consume a sequence number.
  • Both client and server each choose their own independent ISN (x and y).
Arabic — شرح المصافحة الثلاثية
المصافحة الثلاثية (Three-Way Handshake) تُنشئ الاتصال قبل نقل البيانات:
1️⃣ العميل يرسل SYN (مزامنة): "أريد التحدث معك، رقمي الأولي x". يستهلك رقم x.
2️⃣ السيرفر يرد بـ SYN+ACK: "قبلت، رقمي y. وتأكيد استلام SYN (ack=x+1)". يستهلك رقم y.
3️⃣ العميل يرسل ACK: "قبلت. تأكيد SYN السيرفر (ack=y+1)". لا يستهلك رقم (إذا لم يحمل بيانات.
بعد الخطوة 3: الاتصال مفتوح وجاهز لنقل البيانات.
لماذا ثلاث خطوات وليس اثنتين؟ لأن كل طرف يحتاج لتأكيد رقمه الأولي من الطرف الآخر — مصافحتان في اتجاهين مختلفين.

Data Transfer Phase مرحلة نقل البيانات

After the 3-way handshake, data transfer begins. Each segment carries a sequence number and the receiver sends acknowledgments.

Data Transfer Example (Client → Server, ISN=1000, Server ISN=4000)
Client sends 500 bytes: Segment 1: SYN seq=1000 (connection setup, consumes 1) Segment 2: Data seq=1001, data=200B → Server ACKs: ack=1201 Segment 3: Data seq=1201, data=300B → Server ACKs: ack=1501 Server sends 300 bytes: Segment 1: SYN+ACK seq=4000, ack=1001 (consumes 1) Segment 2: Data seq=4001, data=300B → Client ACKs: ack=4301
Arabic
بعد المصافحة الثلاثية تبدأ مرحلة نقل البيانات:
— كل Segment يحمل رقم التسلسل (رقم أول بايت فيه) ورقم التأكيد (رقم البايت التالي المتوقع).

Flow Control — Sliding Window

التحكم في التدفق — النافذة المنزلقة
Flow Control Concept مفهوم التحكم في التدفق

Flow control prevents the sender from overwhelming the receiver's buffer. The receiver advertises a window size — how many bytes it can accept before needing an ACK.

How Sliding Window Works
  1. Receiver tells sender its buffer size (rwnd = Receive Window) in every ACK segment.
  2. Sender can send up to rwnd bytes without waiting for an ACK.
  3. As receiver consumes data (delivers to application), it increases rwnd in subsequent ACKs.
  4. If rwnd = 0: sender STOPS sending and waits for window to open.
Sliding Window Example
Receiver buffer = 1000 bytes. Application reads slowly. ACK 1 from receiver: ack=101, rwnd=1000 → Sender can send bytes 101–1100 ACK 2 from receiver: ack=201, rwnd=800 → Buffer filling up — app not reading fast enough ACK 3 from receiver: ack=201, rwnd=0 → STOP! Buffer full. Sender pauses. ACK 4 from receiver: ack=201, rwnd=500 → Window opens again. Sender resumes. At any time: bytes in flight ≤ rwnd
Arabic — التحكم في التدفق
التحكم في التدفق (Flow Control) يمنع المُرسل من إغراق مخزن المستقبل.
كيف يعمل؟ المستقبل في كل ACK يُخبر المُرسل بحجم المخزن المتاح (rwnd). المُرسل لا يرسل أكثر من rwnd بايت دون انتظار تأكيد.
rwnd = 0: المُرسل يتوقف تماماً ويبدأ بإرسال "Probe Segments" صغيرة بشكل دوري حتى يفتح المستقبل نافذته.
تخيّلها: خرطوم ماء (المُرسل) وسطل (المستقبل). إذا امتلأ السطل → أوقف الماء حتى يُفرّغ السطل.

Error Control & Retransmission

التحكم في الأخطاء وإعادة الإرسال
Error Control Mechanisms آليات التحكم في الأخطاء

TCP ensures reliable delivery through several error control mechanisms:

1. Checksum
Every TCP segment includes a checksum. If the receiver detects a corrupted segment, it silently discards it (no NACK). The sender will retransmit after timeout.
2. Acknowledgment (ACK)
Positive acknowledgment confirms receipt. TCP is cumulative — ACK=N means all bytes up to N-1 have been received correctly.
3. Retransmission Timer (RTO)
After sending a segment, sender starts a timer. If ACK not received before timer expires → retransmit the segment.
4. Out-of-Order Segments
TCP receiver delivers only ordered data to the process. Out-of-order segments are buffered until missing segments arrive.
Normal Operation vs Lost Segment
Normal:
Sender → Seg1 (seq=1)
Sender → Seg2 (seq=2)
Receiver ← ACK (ack=3) — confirms both
Cumulative ACK received
Lost Segment:
Sender → Seg1 (seq=1) ✓
Sender → Seg2 (seq=2) ✗ LOST
Receiver → Dup ACK (ack=2) — "I got 1, need 2"
Receiver → Dup ACK (ack=2) — again
After 3 Dup ACKs → Fast Retransmit Seg2
① The Receiver TCP delivers only ORDERED data to the process
Out-of-order segments are stored in buffer. The process (application) only sees a continuous, ordered byte stream — never sees gaps or duplicates.
Arabic — التحكم في الأخطاء
TCP يضمن الموثوقية عبر أربع آليات:
1️⃣ Checksum: يكشف الأخطاء — إذا تلفت الحزمة تُحذف بصمت والمُرسل يعيد الإرسال بعد انتهاء المؤقت.
2️⃣ ACK التراكمي: ACK=N يعني "استلمت كل شيء حتى البايت N-1". تراكمي = ACK واحد يؤكد عدة Segments.
3️⃣ RTO (مؤقت إعادة الإرسال): إذا انتهى المؤقت بدون ACK → إعادة إرسال. المؤقت يُحسب ديناميكياً بناءً على RTT.
4️⃣ الترتيب: المستقبل يُخزّن الـ Segments غير المُرتّبة في البفر ويُسلّم للتطبيق فقط البيانات المُرتّبة المتسلسلة.
Fast Retransmit: 3 ACK مكررة (Duplicate ACKs) → المُرسل يعيد الإرسال فوراً بدون انتظار المؤقت.

Exam Questions & Complete Answers

أسئلة الامتحان وإجاباتها الكاملة
Part A — Conceptual Questions أسئلة نظرية
Q1 — Fundamental Difference
What is the main difference between UDP and TCP?
Answer: UDP is connectionless and unreliable — no setup, no guarantee, 8-byte fixed header, fast. TCP is connection-oriented and reliable — 3-way handshake, guaranteed ordered delivery, flow & error control, 20–60 byte header. Use UDP when speed matters more than accuracy (VoIP, gaming). Use TCP when all data must arrive correctly (file transfer, HTTP).
Q2 — Transport Layer Role
What is the responsibility of the transport layer?
Answer: The transport layer is responsible for process-to-process delivery — delivering packets from one process (identified by port number) on one host to another process on another host. The network layer handles host-to-host delivery; the transport layer extends this to process-to-process.
Q3 — Port Ranges
A client initiates a connection to a web server. Which port numbers are used on each side?
Answer: Web server uses port 80 (well-known, HTTP). The client OS assigns an ephemeral port (49152–65535) temporarily. The socket pair is: (ClientIP:ephemeral) ↔ (ServerIP:80).
Q4 — SYN/ACK Rules
Does a SYN segment consume a sequence number? Does an ACK segment?
Answer: SYN: YES — always consumes one sequence number even though it carries no data (treated as a virtual byte). SYN+ACK: YES — also consumes one. ACK (no data): NO — a pure ACK with no data consumes no sequence number.
Q5 — Acknowledgment Number
What does the acknowledgment number represent in TCP? Is it cumulative?
Answer: The acknowledgment number represents the number of the next byte the receiver expects to receive. It is cumulative — ACK=1001 means all bytes up to byte 1000 have been received correctly.
Q6 — Ordered Delivery
If TCP receives segment 3 before segment 2, what does it do?
Answer: TCP buffers the out-of-order segment 3 and waits for segment 2. The application receives only ordered data. The receiver may send a Duplicate ACK indicating it still needs segment 2. After segment 2 arrives, both are delivered in order.
Q7 — Flow Control
Describe TCP flow control. What happens when rwnd = 0?
Answer: TCP uses a sliding window. The receiver advertises its available buffer size (rwnd) in each ACK. The sender cannot send more than rwnd bytes without an ACK. When rwnd = 0 the sender stops sending and periodically sends small "probe" segments to check if the window has reopened.
Q8 — UDP Queue Overflow
What happens in UDP when the incoming queue overflows?
Answer: The datagram is silently discarded — no notification is sent to the sender. UDP has no flow control mechanism. This is acceptable for applications like VoIP where retransmitting old audio is worse than losing it.
Part B — Calculation Problems مسائل حسابية
Q9 — UDP Header Decoding
Given UDP header dump: 0045 0035 0026 XXXX
Find: source port, dest port, total length, data length, direction (C→S or S→C)?
Solution:
0045 → Source Port = (0045)₁₆ = 0×16³ + 0×16² + 4×16¹ + 5×16⁰ = 69 decimal 0035 → Dest Port = (0035)₁₆ = 53 decimal ← Well-known port = DNS! 0026 → Total Length = (0026)₁₆ = 2×16¹ + 6×16⁰ = 38 bytes XXXX → Checksum (not asked) Data Length = 38 - 8 = 30 bytes Source port 69 is ephemeral-ish but 53 (DNS) is well-known → Packet: Client → Server (DNS query) Client process: DNS client (port 53 = DNS service)
Q10 — TCP Three-Way Handshake Sequence Numbers
Client ISN = 2000, Server ISN = 5000. Write the 3 handshake segments with all fields.
Solution:
Step 1 — Client → Server: Flags: SYN Seq = 2000 (ISN, consumes this number) ACK = 0 (ACK flag not set yet) Data: none Next seq for client = 2001 Step 2 — Server → Client: Flags: SYN + ACK Seq = 5000 (Server ISN, consumes this number) ACK = 2001 (next byte expected from client = 2000+1) Data: none Next seq for server = 5001 Step 3 — Client → Server: Flags: ACK Seq = 2001 (next after SYN) ACK = 5001 (next byte expected from server = 5000+1) Data: none Seq consumed? NO (pure ACK with no data) Connection ESTABLISHED. Client can now send data starting at seq=2001.
Q11 — Connection Termination
After the connection above, Client seq=2500, Server seq=5300. Write the termination.
Solution:
Step 1 — Client → Server: Flags: FIN Seq = 2500 (consumes 1 seq number) ACK = 5301 (acknowledging all server data) Step 2 — Server → Client: Flags: FIN + ACK Seq = 5300 (consumes 1 seq number) ACK = 2501 (acknowledging FIN = 2500+1) Step 3 — Client → Server: Flags: ACK Seq = 2501 ACK = 5301 (acknowledging server FIN = 5300+1) Seq consumed? NO (pure ACK) Connection CLOSED.
Q12 — UDP Length Formula
An IP packet has Total Length = 512 bytes and IP Header Length = 20 bytes. Is it UDP or TCP? Find UDP data length if applicable.
Solution:
IP payload size = Total IP Length - IP Header = 512 - 20 = 492 bytes If UDP: UDP Total Length = 492 bytes UDP Data = 492 - 8 (UDP header) = 484 bytes If TCP: TCP data = 492 - TCP header (at least 20, up to 60 bytes) If TCP header = 20: Data = 492 - 20 = 472 bytes Note: The protocol (UDP=17, TCP=6) is identified in the IP Header's Protocol field.
Q13 — Sliding Window Problem
The receiver's buffer = 4000 bytes. Application reads at 100 bytes/ms. Sender sends at 500 bytes/ms. How long until rwnd = 0?
Solution:
Net fill rate = Send rate - Read rate = 500 - 100 = 400 bytes/ms Time until full = Buffer size / Net fill rate = 4000 / 400 = 10 ms After 10 ms: rwnd = 0 → Sender MUST stop. After that, application continues reading at 100 bytes/ms (sender stopped adding). Window opens again as application drains the buffer.
